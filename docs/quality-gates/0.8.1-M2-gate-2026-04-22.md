# CERBERUS 0.8.1 M2 quality gate — 2026-04-22

M2 closes the research-gap cache-depth items from the 0.8.1 plan: L1 pointer-chase latency probe (research gap A), L2 reach via an ephemeral 64 KB FAR buffer (research gap B, narrowed scope), and DRAM ns derivation (research gap E). The 128 KB / 256 KB sweep envelope from the original plan was trimmed to 64 KB and the rest deferred to 0.9.0; see the scope discussion below.

## Gate outcome

**PASS**. Zero BLOCK, one WARN carried from M1/M3 (DGROUP AT RISK yellow zone), one WARN new (64 KB-only L2 reach leaves the >64 KB cache case undetected until 0.9.0 huge-pointer work). No regressions; all previously-landed tests continue to pass.

## M2 tasks landed

| Task | Status | Surface |
|---|---|---|
| M2.1 L1 pointer-chase latency probe | **Implemented** | `src/bench/bench_cache_char.c` gains `pointer_chase_latency_ns()`. Reinterprets the existing 2 KB shared small buffer as a 1024-slot `unsigned int __far` array, initializes it with a coprime-step (67) chain that touches every slot before repeating, primes the cache with one full cycle, then measures 20,000 data-dependent chase iterations via PIT-C2. | M2.2 L2 cache detection | **Narrowed to single-step 64 KB** | `_fmalloc(65520)` gets an ephemeral 64 KB FAR buffer, `_fmemset` zeroes it, `stride_read` measures throughput, `_ffree` releases. ## WARN items

### W14 (WARN, carried from M1/M3): DGROUP AT RISK

Headroom vs 62 KB soft target continues to shrink slowly: M1 left 2,016 bytes, M3 1,824, M2 1,664. The M2 delta (+160 bytes) is well under the 500-byte cap from the plan; the AT RISK status continues to reflect accumulated pressure from prior milestones.

Before the v0.8.1 tag the plan recommends a CONST-segment audit to recover headroom. With M2 landed this becomes actionable since 0.8.1's code surface is now fully in the binary.

Recommend: run `wmake dgroup-report` + a grep for duplicate string literals across `src/` before tagging. Historical wins have come from consolidating per-key format strings.

### W17 (WARN, new): M2.2 L2 reach is 64 KB-only; 128 KB / 256 KB deferred

The original 0.8.1 plan targeted a 64 / 128 / 256 KB sweep. The implemented version measures only 64 KB because `_fmalloc` in Watcom medium model is capped at ~64 KB per call and larger contiguous FAR buffers require `halloc` + huge-pointer arithmetic, which in turn requires a huge-pointer version of `stride_read` we do not yet have.

What ships: the 64 KB measurement answers the most common 486-era question ("is there an L2 reachable from a 64 KB working set?") by direct comparison with the existing 32 KB sweep. A consistent throughput indicates cache coverage past L1; a drop indicates DRAM reach.

What does not ship: distinguishing L2 sizes >64 KB. On a system with 256 KB L2 (common on later Pentium boards), the 64 KB measurement still lands in L2 and the emitted `dram_ns` is an L2 rate mis-labeled as DRAM. Confidence on `dram_ns` is already MEDIUM to acknowledge this.

Recommend: file 0.9.0 task to add huge-pointer `stride_read_huge()` + `halloc`-backed 128 KB / 256 KB buffers. At that point `bench_cc_derive_dram_ns()` can confidently pair the largest-successful sweep with the line-size and report true DRAM ns.
