## DOSBox-X empirical findings (detect-path smoketest, M2 probes)

- `fpu.fptan_pushes_one=yes` — correct (DOSBox-X 486 emulated FPU is 387+ era)
- `fpu.rounding_modes_ok=yes` — all 4 RC modes match IEEE-754 canonical table on emulation. Per-mode: nearest `(2,-2)`, down `(1,-2)`, up `(2,-1)`, truncate `(1,-1)`.
- `fpu.precision_modes_ok=no` — **real probe finding**. DOSBox-X's FPU emulation doesn't honor the PC (precision-control) field: all three PC modes produce byte-identical 1.0/3.0 extended results. This is a known emulation gap, flagged by the probe working as designed. Real iron expected to show `yes`.
- `fpu.exceptions_raised=0_of_6` — **real probe finding**. DOSBox-X doesn't raise exception flags in the status word. This is a known emulation gap. Real iron expected to show `6_of_6`.

These two `no`s are the M2 story: finer-grained FPU characterization surfaces emulation vs hardware differences that earlier passes couldn't see. Tomorrow's real-iron captures (486 BEK-V409 plus 386 DX-40) will be the first data points showing what real hardware does for these axes.

## WARN items carried forward

### W7 (WARN): M2.4 precision-control emits summary only, not per-mode detail

Per-mode 10-byte tword results of 1.0/3.0 under PC=single/double/extended are computed and used by the inference function but NOT emitted as INI rows. Rationale: each would be ~20 hex chars + key label, 3 rows × ~35 bytes = 105 bytes DGROUP for the display buffers alone. Preserved for M2 extension pass if real-iron shows `precision_modes_ok=no` on a CPU where it should be `yes`.

### W8 (WARN): M2.6 exception probe emits compact count only, not per-exception detail

Per-exception status-word captures are computed but not emitted as INI rows (same DGROUP rationale). Real-iron inspection will tell us if any one exception is silent — if so, expand the emit.
