## Round 1: Silent failures / trust-damage risks

### W1 (WARN): Mandelbrot coda removed from stock builds

Scope: regression of visual journey.

`bench_whetstone.c` places `bench_mandelbrot_demo(o)` inside the
`#else CERBERUS_WHETSTONE_ENABLED` branch (line 273). Stock builds
early-return before reaching it. Previously the Mandelbrot ran
whenever an FPU was present as "post-run proof-of-life for the FPU"

Impact: one visual beat removed from stock run. `bench_fpu.c`
throughput + FPU behavioral fingerprint (v0.7.1) cover the FPU
liveness story without needing Mandelbrot, so the narrative justifies
the cut. Tony's real-iron run confirmed no Mandelbrot render, and the
journey still feels complete (visual flow: intro → bit_parade →
lissajous → cache_waterfall → latency_map → metronome → audio_scale).

Recommend: accept for M1. If Tony wants Mandelbrot back as an
independent FPU visual in stock, M2 could refactor it out of
`bench_whetstone.c` into `bench_fpu.c`'s tail. Low-cost fix; out of
scope for M1.

### W2 (WARN): Research-build `upload.status` stale on offline/skipped/no_client paths

Scope: research-build only bug my M1 work exposed.

`upload_populate_metadata` seeds `upload.status=not_built` in the
table, then `report_write_ini` flushes to disk. In research builds
(`wmake UPLOAD=1`), `upload_execute` then updates status in memory to
`offline` / `skipped` / `no_client`, but the offline/skipped/no_client
paths do NOT call `report_write_ini` a second time. The INI on disk
keeps `status=not_built` while the UI sees `skipped`. Verified today
on DOSBox-X with research build + `/NOUPLOAD`: UI showed "skipped",
disk INI showed "not_built".

Impact: stock builds are correct (status=not_built is accurate; no
stale state). Research builds have misleading INI values on those
three paths.

Recommend: fix as part of M1 closure — three 1-line additions in the
research `upload_execute` paths. See action item A2 below.

### W3 (WARN): Em-dashes in source comments (preference violation)

Scope: code hygiene, not user-visible.

Tony's memory explicitly says: "Tony detests em-dashes and
sentence-punctuation hyphens (incl. `--`). Use commas, colons,
semicolons, periods." My M1 work added em-dashes in comments across
`src/bench/bench_whetstone.c`, `src/upload/upload.c`, `src/detect/cpu.c`,
`src/main.c`, `Makefile`, `tests/host/test_consist.c`,
`docs/methodology.md`, `docs/ini-format.md`, and `tests/target/
VALIDATION-0.8.0-M1.md`. Comments do not compile to binary, so no
runtime impact, but the preference applies broadly.

Runtime-visible em-dashes (string literals) have already been purged.
The current deployed binary has zero em-dash bytes (`0xE2 0x80 0x94`
triplets) confirmed via `grep -c`.

Recommend: sweep my M1 additions for em-dashes in a separate style-
pass. Out of scope for M1 trust-first gate; carry to M4 (docs parity).

### N1 (NOTE): `_nmalloc` fallback untested in production

`upload_populate_metadata` falls back to static `upload_nickname_fallback` /
`upload_notes_fallback` when `_nmalloc` returns NULL. This fallback
was designed but has never executed in any captured run (heap is
never exhausted on any tested machine including BEK-V409 with
HIMEM+EMM386). Code is straightforward; trusting it unexecuted is
acceptable, but a host test that forces the NULL path to run through
the fallback would be cheap belt-and-braces.

Recommend: defer to M2.

## Round 2: Fix regressions

### W4 (WARN): BIOS date `[bios] dree=�    92` BSS stomp on real iron

Scope: data corruption in INI, pre-existing overwriter exposed by M1.3
BSS layout shift.

BEK-V409 real-iron INI shows `[bios] dree=�    92` where expected was
`date=11/11/92`. The 4-byte key "date" got stomped to "dree" and the
value got partial garbage. Same class of bug as issue #9 (nickname
leak) but different victim. My M1.3 fix moved `upload_nickname_buf`
out of BSS via `_nmalloc`, which shifted BSS layout enough that the
unknown overwriter now targets `bios.date`'s key or the buffer
holding "11/11/92".

Impact: cosmetic in INI (human reader can still recognize the section
context); does NOT affect any consistency rule (rules don't check
bios.date format); does NOT affect upload (only stock emits
not_built anyway). The underlying overwriter is the real issue — an
unknown buffer somewhere in detect/diag/bench writes past its bounds
into a BSS neighbor.

DOSBox-X does NOT reproduce the stomp (capture shows
`date=01/01/92` clean). This is real-hardware-specific, likely due
to different memory allocation patterns.

Recommend: track as KNOWN ISSUE to investigate in M2. Do not block
M1 tag. The overwriter has been here since at least v0.7.1; finding
it requires methodical diff-rebuild-capture cycles on real hardware
and is out of scope for M1's "ship cleanly" identity.

### N2 (NOTE): Rule 10 effectively dormant in stock builds

## Post-gate updates (2026-04-21 20:35)

### End-of-run hang resolved (closes N5 partially)

BEK-V409 real-iron capture with M1 binary (post-em-dash-fix) ran
cleanly through the scrollable summary, Q-exit, and the post-UI
crumb chain. `CERBERUS.LAS` retained `main.return` on the hung
run, definitively localizing the hang to Watcom's libc exit
teardown (atexit chain / FPU state / stdio close) AFTER `return
exit_code` executed.

**Fix applied**: `src/main.c` replaces the final `return exit_code`
with a `crumb_exit()` (to unlink LAS on successful exit) followed
by `_exit((int)exit_code)` which calls INT 21h AH=4Ch directly and
bypasses libc cleanup. All resources (stdio, video, open files)
are released by explicit calls before this point; CERBERUS has
zero registered atexit handlers. The libc cleanup was adding no
value and was empirically hanging on BEK-V409 under DOS 6.22 +
HIMEM + EMM386.

Verified 2026-04-21 20:39: BEK-V409 exits cleanly to DOS prompt
after Q-exit. `CERBERUS.LAS` absent post-run. INI complete. All
M1.1-M1.5 invariants present in the capture
(`tests/captures/486-BEK-V409-2026-04-21-m1-exit/capture.ini`).

### W6 (WARN, post-gate discovery): Watcom NULL assignment check fires at exit

With the hang fixed and exit path now reaching libc's end-of-run
check, Watcom's "*** NULL assignment detected" message emits to
the console on every BEK-V409 run. This is Watcom's built-in
canary: a signature pattern in the first 32 bytes of DGROUP
(`_NULL` segment), verified at program exit. Corruption of those
bytes indicates a NULL (or near-NULL) pointer write somewhere in
the program's lifetime.

**Evidence unifying this with W4**: the BIOS date stomp (`dree=`
byte pattern in `[bios]` section) is CONST-segment corruption;
CONST lives at DGROUP offset 32 (immediately after `_NULL`). A
write that corrupts `_NULL`'s 32 bytes AND spills past the
boundary into CONST explains both symptoms with a single
underlying defect. On 2026-04-21 20:39's clean-INI run, the stomp
landed only within `_NULL` (no CONST damage this time) because
the OPL probe succeeded (`opl=opl3` path, shorter probe trace).
Prior run with OPL fallback (`opl=none`, longer probe trace) saw
the stomp overflow into CONST, corrupting the BIOS date string.

DOSBox-X does NOT reproduce the NULL detection (verified same
session: DOSBox-X capture shows clean INI and no NULL message).
Real-hardware-specific, same investigation class as W4.

**Recommend**: track as a single M2 investigation item covering
both symptoms. The NULL-assignment detection gives a reliable
trigger for the bug (every BEK-V409 run fires it, so it's
deterministic on real iron). The BIOS date stomp gives a
path-dependent variant (triggers only on certain detect-phase
paths). Combined, an M2 session can isolate via removal-at-a-time
on BEK-V409.

**Mitigation for 0.8.0 release messaging**: do NOT suppress
Watcom's check. The message is evidence of a real defect;
silencing it would be trust-damaging. Instead, document openly
in release notes + README that CERBERUS 0.8.0 has a known
real-iron NULL-assignment bug that Watcom's runtime detects and
reports; fix is tracked for 0.8.1 or M2.
