# CERBERUS v0.4.0, Benchmarks and Polish (final)

**Final release.** Tagged on 2026-04-19. Closes the UI defects found in v0.4-rc1's BEK-V409 screenshot. The rest of the rc1 envelope carries forward unchanged. Fourth tag in the weekend arc after `v0.2-rc1`, `v0.3-rc1`, `v0.4-rc1`.

## What changed since v0.4-rc1

Three targeted fixes between `ab67703` (rc1) and the v0.4.0 tag. All three are visible in the validation capture at `tests/captures/486-real-2026-04-19-v0.4.0/`.

### Fix 1. `diag_cache` and `consist` status string CP437 corruption

The rc1 BEK-V409 screenshot rendered `cache.status` as:

```
[PASS] cache.status    ratio=1.69 ╢ù ╔╒╓ cache_working
```

Root cause in `src/diag/diag_cache.c:300`. The sprintf format string contained U+00D7 MULTIPLICATION SIGN (UTF-8 `0xC3 0x97`) and U+2014 EM DASH (UTF-8 `0xE2 0x80 0x94`). Those multi-byte sequences reach the UI pane verbatim and DOS's text-mode VRAM renders each byte as its CP437 glyph, producing garbage between the ratio value and the status tag.

Fix: ASCII-only format string, `"%s (ratio=%lu.%02lu x - %s)"`. The same audit was applied preemptively to `src/core/consist.c`'s Rule 4b cache-contextualized warnings (two paths) and Rule 11's XT-class DMA contradiction warning. All three emit to the SYSTEM VERDICTS pane and contained the same UTF-8 em-dash latent bug pattern. The rc1 BEK-V409 run did not trigger those paths (hardware was 486, not XT-class; bench range was normal), so they did not surface in the rc1 screenshot. They would have corrupted on different hardware or under load.

The v0.4.0 BEK-V409 validation triggered both `cache.status` (clean) and Rule 4b's cache-contextualized warning (line 148 of the validation INI: `cpu_ipc_bench=WARN: bench 1964636/sec below expected 4700000 (cache diag PASS - TSR stealing cycles or thermal throttle)`). Both rendered clean ASCII, confirming the preemptive fix earned its keep.

Landed at commit `2d6a0a7`.

### Fix 2. UI `value_str` type-aware rendering

The rc1 screenshot showed five BENCHMARKS rows rendering labels but blank values:

- `fpu ops/s`
- `mem write`, `mem read`, `mem copy`
- `k-whet (LOW)`

All five are V_U32 records emitted with `display=NULL` (the anti-DCE pattern from commit `8552c6d`). The INI emit path formats these correctly via `format_result_value` at `src/core/report.c:235`. `src/core/ui.c`'s `value_str` returned `""` for any row where `display==NULL` and `type != V_STR`, so labels rendered while values did not.

Fix: mirror `format_result_value`'s switch inside `value_str`. Uses a 32-byte static scratch buffer; the return pointer is valid until the next `value_str` call. The invalidation constraint is documented in the function comment so future refactors do not trip it. Current callers all consume the pointer before re-entering, so no semantic change ships with this commit, only the value-to-string conversion that was missing.

Detection pane rows are V_STR only, so they were unaffected and remain unchanged. The title bar is also V_STR (`cpu.detected`, `video.chipset`, `bios.date`). Only the BENCHMARKS pane needed the type-aware path, and all five previously blank rows now render correctly.

The v0.4.0 BEK-V409 validation screenshot shows:
- `fpu ops/s   1118568`
- `mem write   15503`, `mem read   16260`, `mem copy   7220`
- `k-whet (LOW) 109`

Quality gate: 3 rounds self-applied, capped per the bench_video cap-at-3 lesson from the rc1 cycle. R1 verified correctness (logic mirrors `format_result_value` verbatim). R2 considered exposing `format_result_value` publicly but rejected for blast-radius reasons. R3 characterized the diff as 31 lines, one file changed, no regression to the detection pane or title bar.

Landed at commit `841d7c3`.

### Fix 3. `bench_dhrystone.c` comment updated per Phase 2 T2 lesson

Comment-only, not visible to users. Reflects the Phase 2 Homage research finding: CheckIt 3.0's "Dhrystones" is a custom synthetic using the familiar label, not a Dhrystone 1.1 or 2.1 port. Zero canonical Weicker markers in `CHECKIT.EXE` or `CHECKIT.OVL`. The 33,609 BEK-V409 anchor and ±5% tolerance remain unchanged as the empirical match target. The comment now frames it correctly as an empirical agreement point, not an algorithmic equivalence.

Full derivation at `docs/research/homage/checkit-dhrystone-version.md`.

Landed at commit `6acb559`.

## What validated

### BEK-V409 real-hardware capture

Intel i486DX-2-66, AMI BIOS 11/11/92, S3 Trio64, Vibra 16S, 63 MB XMS. Full capture archived at `tests/captures/486-real-2026-04-19-v0.4.0/CERBERUS.INI`. Four photographs of the run at `docs/releases/v0.4.0/screenshots/`:

- **`three-pane-ui-closeup.jpg`**. The three-pane UI at the moment before the `C:\CERBERUS>` prompt returns. `[PASS] cache.status ratio=1.48 x - cache_working` renders clean ASCII; all five previously blank BENCHMARK rows carry values.
- **`three-pane-ui-bench-box.jpg`**. Same UI, wider frame showing the open 486 case on the right.
- **`bench_video-text-mode.jpg`**. `bench_video` mid-run writing through the full CP437 color-attribute space to the text-mode segment. Every cell gets a character and attribute combination in sequence. What the screen shows is the VRAM-mapped render of the write loop in progress. This is the benchmark itself, visualized.
- **`bench_video-mode13h.jpg`**. `bench_video` in mode 13h (320×200×256), writing horizontal stripes of palette data. Again, the on-screen pattern is the benchmark's VRAM write loop rendered live. The mode exits cleanly back to mode 3 after measurement.

### Host tests

163 assertions across timing, consist, thermal, diag_fpu, diag_cache, diag_dma, bench_cache. All green except the 4 pre-existing `test_timing` failures (issue #1, gated behind the UMC491 8254 phantom-wrap deep-dive, unchanged since v0.2-rc1).

## What did not ship (still)

- **Issue #4. Whetstone FPU-assembly rework.** `bench.fpu.k_whetstones` stays CONF_LOW at ~109 K-Whet (CheckIt reference 11,420, ~100× low). Unchanged from rc1.
- **Issue #6. `bench_video` measures ISA-range bandwidth on VLB hardware.** The v0.4.0 capture showed `bench.video.mode13h_kbps = 4,613` on the BEK-V409 Trio64 VLB system. Investigation plan stands. Homage Phase 2 FASTVID research (`docs/research/homage/fastvid-mtrr-reference.md`) confirmed the original tool-misidentification correction but did not change the issue #6 technical posture.
