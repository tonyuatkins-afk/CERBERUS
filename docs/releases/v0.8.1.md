# CERBERUS v0.8.1, Completion Release

Tagged on 2026-04-22, one day after v0.8.0. 0.8.1 is the completion release for the items explicitly deferred from v0.8.0 into 0.8.1 per the v0.8.0 release notes. Scope is smaller than v0.8.0: no new milestones, no scope expansion, the trust-first cuts from 0.8.0 (Whetstone gate, upload compile-out) stay in place.

## Headline

Every item the v0.8.0 release notes marked "deferred to 0.8.1" is either landed or explicitly carried forward with a documented reason. The v0.8.0 trust-first filter ("real-iron-proven or do not ship") holds: M4 BEK-V409 BSS overwrite root-cause is hardware-gated and continues as an open item for the next bench-box session.

## What shipped

### M1. Precision and output completion

- **IEEE-754 edge-case diagnostic** (M1.1, research gap L). 14 focused edge cases across FADD (4) / FSUB (1) / FMUL (2) / FDIV (3) / FSQRT (4) exercise the documented IEEE-754 corners: signed zero preservation, infinity arithmetic, NaN propagation, 0/0, inf/inf, sqrt(-0). Per-op counters emitted as `diagnose.fpu.edge.<op>_ok=N_of_M`, aggregate as `diagnose.fpu.edge_cases_ok=14_of_14` on a conformant FPU. FSQRT uses `#pragma aux fsqrt` to emit the bare instruction instead of libc `sqrt()` (Watcom's libc clamps domain-error inputs to 0.0 rather than propagating NaN per IEEE-754). SNaN is not exercised: pre-387 silent-accept is already covered by `diag_fpu_fingerprint`. DOSBox Staging smoketest on emulated 486: 14/14 pass. (`src/diag/diag_fpu_edges.c`)
- **`/CSV` output mode** (M1.2, research gap H). When set, writes a sibling `.CSV` file alongside the INI with RFC 4180 minimal quoting (fields containing comma, quote, CR, or LF get quoted with embedded quotes doubled). Schema: `key,value,confidence,verdict` header + one row per result. Confidence renders as `HIGH`/`MEDIUM`/`LOW`; verdict as `PASS`/`WARN`/`FAIL`/`UNK`. (`src/core/report.c`)

### M2. Cache depth expansion

- **L1 pointer-chase latency probe** (M2.1, research gap A). Reinterprets the existing 2 KB shared FAR buffer as a 1024-slot `unsigned int` array, initializes with a coprime-step (67) chain that visits every slot before repeating, primes the cache with one full cycle, then measures 20,000 data-dependent chase iterations via PIT-C2. Emits `bench.cache.char.l1_ns`. At single-digit ns resolution for aggregate-timed bursts of 0.3-0.6 ms on a 486-class target. (`src/bench/bench_cache_char.c`)
- **L2 reach via 64 KB FAR buffer** (M2.2, research gap B, narrowed scope). `_fmalloc` allocates an ephemeral 64 KB buffer, `stride_read` measures throughput at a 64 KB working set, `_ffree` releases. Emits `bench.cache.char.size_64kb_kbps` + `bench.cache.char.l2_status=ok`. On allocation failure, emits `l2_status=no_far_mem` without disturbing the run. **Scope-cut**: the original 64 KB / 128 KB / 256 KB envelope from the 0.8.1 plan was narrowed to 64 KB only. Larger working sets require huge-pointer arithmetic to span the 64 KB segment boundary, deferred to 0.9.0. What ships answers the common 486-era "is there L2 reachable from a 64 KB working set" question.
- **DRAM ns derivation** (M2.3, research gap E). New pure helper `bench_cc_derive_dram_ns(kbps, line_bytes)` returns `line_bytes * 1e6 / kbps`. Emits `bench.cache.char.dram_ns` at `CONF_MEDIUM` (the 64 KB working set may still land in L2 on chips with larger L2, an ambiguity the 0.9.0 larger-sweep work resolves). Fully host-testable.

### M3. DB and detection gap closure

- **IIT 3C87 FPU routing** (M3.1, partial). `hw_db/fpus.csv` gains the `iit-3c87` row (IIT vendor, matrix-math note); `src/detect/fpu_db.c` regenerated with 15 entries. `src/detect/fpu.c` adds `fpu_probe_iit_3c87()` as a documented stub returning 0, with a header comment enumerating the two documented discrimination approaches (FNSAVE extended-bytes check, or opcode probe via INT 6 handler). Routing is in place so the real probe is a small-diff edit once a 386 DX-40 + IIT 3C87 real-iron capture provides the exact signature. **Hardware-gated remainder:** the actual IIT vs Intel 80387 discriminator is one of the real-iron items tagged open for the next bench session.
- **Genoa ET4000 chip-level probe** (M3.2, implemented, awaiting real-iron validation). `src/detect/video.c` adds `probe_et4000_chipid()` — a segment-select-register (3CDh) read-write-readback test that fires on any ET4000-family chip regardless of BIOS string. Integrated into `detect_video` orchestration after `probe_s3_chipid()`, before the BIOS ROM text scan. No new DB row; Genoa OEM'd Tseng silicon, so the existing ET4000 row is authoritative. The 3CDh behavior is sourced from the Tseng programmer's reference; confirmation awaits the real-iron pass on the 386 DX-40 + Genoa box.
- **Hercules variant discrimination** (M3.3, implemented). `src/core/display.h` grows a `hercules_variant_t` enum (HGC / HGC+ / InColor / UNKNOWN / NA). The pure classifier lives in a dedicated include (`src/core/display_hercules_ids.c`) so the host test can exercise it without pulling in DOS-only headers. `display_hercules_variant()` samples 3BAh outside vertical retrace and reads bits 6:4 as the 3-bit variant ID. `video.hercules_variant` emits alongside the stable `video.adapter=hercules` signature key. 16-assertion host-test suite covers the documented IDs, undocumented bit patterns bucketing as UNKNOWN, high-bits masking, and token mapping.

### Hotfix

Between the M2 commit and the smoketest, the DOSBox Staging end-to-end pass surfaced a `/CSV` flag regression: `str_starts_with(a, "/C")` was consuming `/CSV` as the `/C` calibrated-mode prefix before the exact `/CSV` match branch was reached, so `do_csv` never got set and the CSV file was never written. Fix: moved the `/CSV` exact match immediately after `/Q` and tightened the `/C` prefix guard to require `a[2]` be NUL or colon.

This bug passed both the host-test suite and the initial `wmake` build; only the DOSBox-Staging end-to-end exposed it. The lesson is documented as a durable instruction: after every CERBERUS.EXE build, run the DOSBox Staging smoketest before claiming the binary clean. Host-test green plus `wmake rc=0` is not sufficient proof.

## What did not ship (carried to 0.8.2 or 0.9.0)

**To 0.8.2 (hardware-gated):**
- **M4 BEK-V409 BSS overwrite root-cause**. The defect only reproduces on the BEK-V409 486 DX-2-66 + AMI 11/11/92 + DOS 6.22 + HIMEM + EMM386 stack and requires a dedicated debugging session on that box with the removal-at-a-time probe-path protocol documented in `docs/methodology.md`.
- **M3.1 IIT 3C87 real discriminator**. The DB row and routing are in place; the signature (FNSAVE extended bytes or opcode fault) becomes a small edit once a 386 DX-40 + IIT 3C87 capture supplies it.
- **M3.2 Genoa ET4000 probe confirmation**. Algorithm is from documentation; real-iron validation on the 386 DX-40 + Genoa board confirms it fires correctly.

**To 0.9.0:**
- **L2 sweep at 128 KB / 256 KB working sets**. Requires huge-pointer `stride_read_huge()` + `halloc`-backed buffers to span 64 KB segment boundaries.
- **Per-instruction FPU microbenchmarks** (research gap N). Whetstone replacement story.
- **Upload path re-enabled** once the server is live and a stack-safe offline fallback is shipped.
- **8088/XT real-hardware capture**. README claim retraction holds.
- **Full CUA shell** (menus, dropdowns, modal dialogs).
- **Hardware identification expansion**: BIOS ROM 64 KB hash, expansion ROM walk (C800:0000 to DFFF:FFFF), network card + OUI detection, UART FIFO type, parallel ECP/EPP, game port, RTC presence. Scope brief filed at `project_cerberus_0_9_0_hw_id_roadmap.md` in the auto-memory.

## Validation

### Smoketest

DOSBox Staging smoketest on the tagged 0.8.1 binary (486 + 16 MB + Sound Blaster 16 + S3 Trio64 profile):
- **M1.1**: `diagnose.fpu.edge_cases_ok=14_of_14,HIGH,PASS` — every op passed on the emulated 486 FPU.
- **M1.2**: `devenv/SMOKE081.CSV` emits a 102-row file with correct RFC 4180 header and rows.
- **M2.1/M2.2/M2.3**: `bench_cache_char` self-skips under emulators by design (probe shape unreliable on host-cached guest); these are real-iron-only and validated by host tests for the pure inference helpers.
- **M3.2**: `probe_et4000_chipid` ran on Staging's S3 emulation and correctly returned NULL (not an ET4000); the S3 Trio64 probe won, as expected.
- **M3.3**: `video.hercules_variant` absent (Staging reports as VGA, not Hercules) — the gating is correct.
- **Flag parsing regression check**: `mode=quick, runs=1` on `/CSV` invocation confirms the hotfix.

### Host tests

**376 assertions** across **12 executables**, 0 failures. Up from v0.8.0's 320 / 9. Added suites: `test_diag_fpu_edges` (21 assertions), `test_report_csv` (15 assertions), `test_hercules_variant` (16 assertions); plus DRAM-ns additions in `test_bench_cache_char`.

### Real-iron

No real-iron capture for 0.8.1. The items hardware-gated above are the explicit open real-iron work. Every new probe in 0.8.1 either (a) runs deterministically regardless of hardware (M1.1, M1.2, M3.3), (b) self-skips under emulators (M2.*), or (c) is a documented stub (M3.1) / documented-but-unvalidated probe (M3.2) awaiting the next real-iron session.
