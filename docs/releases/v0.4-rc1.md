## What shipped

### bench_cache — L1 + RAM throughput benchmark

New module `src/bench/bench_cache.c` emits four KB/s throughput rows plus elapsed-us partners and a composite checksum:

```
bench.cache.small_write_kbps   stride-byte writes over 2 KB buffer
bench.cache.small_read_kbps    stride-byte reads over 2 KB buffer
bench.cache.large_write_kbps   stride-byte writes over 32 KB buffer
bench.cache.large_read_kbps    stride-byte reads over 32 KB buffer
bench.cache.checksum           read-sink | write-sink, 16-bit signal
```

Shares FAR buffer storage with `diag_cache` via a new `src/core/cache_buffers.{c,h}` module — the 2 KB + 32 KB allocation is single-sourced across both consumers, with an explicit `cache_buffers_reset()` helper establishing the inter-caller ownership protocol. Compiles at `CFLAGS_NOOPT` (-od -oi) to match the Dhrystone / Whetstone DCE-safety posture. Uses `timing_start_long` / `timing_stop_long` for BIOS-tick-resolution multi-second measurements.

**Keys are size-based, not cache-aware.** The `small_*` / `large_*` framing deliberately replaces an earlier `l1_*` / `ram_*` naming because write-through 486 caches push every store through to DRAM regardless of cache state — `small_write ≈ large_write` on that hardware, and the L1/RAM labels would have misled. Consumers cross-reference the four rates to infer cache policy.

### bench_video — text-mode and mode 13h VRAM throughput

New module `src/bench/bench_video.c` emits two KB/s measurements with status partners:

```
bench.video.text_write_kbps    stride writes to detected text segment
                                (B800 modes 0/1/2/3, B000 mode 7)
bench.video.mode13h_kbps       stride writes to A000:0000 during mode 13h
                                (VGA color / MCGA only, skipped on emulator)
```

Queries current video mode via `INT 10h AH=0Fh` at entry (not adapter class), so a user sitting in mode 7 (mono) on an EGA-color adapter gets the right segment. Supports mode 0/1 (40x25, 0x0800 page stride) and 2/3/7 (80x25, 0x1000 stride). Computes VRAM offset from `active_page * page_stride` so multi-page text modes get their live display measured and restored. Save/restore preserves the user's display; a warning `puts` before the ~1 second VRAM garble phase tells them to expect the flicker. Mode 13h round-trip restores the exact entry mode (masking bit 7 of the AL return from AH=0Fh to avoid honoring a stray "no-clear VRAM" flag).

Gates mode 13h on `ADAPTER_VGA_COLOR` or `ADAPTER_MCGA` AND `environment.emulator=none` — mode switches under DOSBox-X / emulators produce unreliable timings. Fail-safe skip (`skipped_detect_not_run`) when `/ONLY:BENCH` bypassed detect.

### Rule 4b narration extension

`rule_cpu_ipc_bench` now consults `diagnose.cache.status` when bench_cpu reports under-range iters/sec. Three-way narration split:

- Cache status contains `cache_working` → "cache diag PASS — TSR stealing cycles or thermal throttle" (cache exonerated)
- Cache status contains `no_cache_effect` → "cache diag FAIL — cache disabled in BIOS or absent" (TSR exonerated)
- Cache status missing or inconclusive → original three-cause ambiguous narration (no new evidence to prefer one cause)

No new rule number. Reuses `msg_rule4b_warn` buffer. Five new test_consist scenarios (JJ-NN) cover all three branches plus in-range PASS and over-range WARN with narration-substring assertions.

### Rule 11 `dma_class_coherence`

XT-class CPUs (8086/8088/NEC V20/V30) shipped with only the master 8237 DMA controller. Rule 11 cross-checks `cpu.class` against `diagnose.dma.ch5_status`:

- XT + `skipped_no_slave` → PASS (coherent)
- XT + `pass` (slave responsive) → WARN (detection contradiction)
- XT + `fail` → no-op (per-channel row already surfaces)
- AT-class or missing keys → no-op

Catches misidentified CPUs being reported as XT-class when the box is AT, DMA readback false positives on a floating ch5 port, and frankenboard XT + retrofit slave DMA. Eight new test_consist scenarios (OO-WW) cover all branches.

## What didn't ship

- **Issue #4: Whetstone FPU-assembly rework.** `bench.fpu.k_whetstones` stays CONF_LOW at ~109 K-Whet (CheckIt reference 11,420 — ~100× low). Closing this requires NASM-compiled x87 inner kernels for `PA()`, `P0()`, and Module 2's hot loop with Watcom+NASM calling-convention work. Scoped for a dedicated session with fresh context; x87-assembly interop is new territory for CERBERUS.

- **Auto-calibrating bench_cache / bench_video iteration counts.** Current counts are tuned for 486 DX-2-66 center. Faster hardware (AMD 5x86, DX-4-100, VLB VGA) may drop below the 10-BIOS-tick resolution floor. Fixed iters are the v0.4 posture; auto-calibration deferred to v0.5 with the bar-graph comparison UI.

- **Shared `find_local` helper.** Three copies across `bench_cache.c`, `bench_video.c`, `bench_all.c`. Consolidation into `report.h` is v0.5 refactor scope.

## Measurement-scope caveats

Both bench_cache and bench_video emit absolute KB/s numbers, but:

- **bench_cache rates are NOT cross-tool comparable** to CheckIt, CACHECHK, or SPEEDSYS numbers on the same hardware. `CFLAGS_NOOPT` pins the inner byte-stride loops at unoptimized C, which trades absolute throughput for DCE safety. Same-machine run-over-run regression detection and thermal tracking remain useful; direct cross-tool scale comparisons do not.
- **bench_video rates reflect C-loop VRAM write throughput, not peak bus bandwidth.** See issue #6 for the VLB-range investigation scope.

The same methodological posture governs bench_dhrystone (~2% of CheckIt reference) and bench_whetstone (~1% — issue #4).
