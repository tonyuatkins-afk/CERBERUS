## Design: `diag_cache`

### Intent

Verify that the CPU's L1 data cache (on 486+) is present, enabled, and coherent. Catches:

- **Cache disabled in BIOS.** Common v0.2 rc observation on the BEK-V409 bench box: `cpu_ipc_bench` WARN at 1.96M iters/sec vs expected 4.7M — symptom of cache disabled. This diagnostic confirms the cause structurally.
- **Counterfeit/remarked CPUs where cache is absent but CPUID claims present.** Rare but real in the DOS vintage scene.
- **Intermittent cache failures** (e.g., aging capacitor near cache RAM, dusty heatsink causing thermal dropout) — run-to-run variance in the diagnostic indicates instability.

### Does NOT detect

- Cache SIZE or associativity. Those are detection-module concerns (`detect_cache`, not `diag_cache`).
- Cache coherence under DMA/bus-master activity. Would need a contrived test setup — out of scope for v0.3.
- External L2 cache (motherboard SRAM). The timing signal for internal vs external cache is similar and v0.3 doesn't disambiguate; the diagnostic reports "cache present and working" if ANY level produces the signal.

### Method: stride-ratio timing

Classic DOS-era cache-presence test, used by CheckIt, SpeedSys, and similar:

1. Allocate a buffer **smaller than expected L1** (e.g., 2 KB — 486 has 8 KB L1, so 2 KB fits comfortably).
2. Time N iterations of a read loop over the small buffer with stride 4 bytes. Call this `t_small`.
3. Allocate a buffer **much larger than expected L1** (e.g., 128 KB — well above any plausible on-die L1 for 486 family).
4. Time N iterations of the same read-loop pattern over the large buffer. Call this `t_large`.
5. Compute ratio `t_large / t_small`. **Interpretation:**
   - **Ratio ≈ 1.0** — both reads hit the same cache (somehow). Unlikely on real hardware unless cache is much bigger than either test buffer. Report WARN: "cache unusually large or measurement ambiguous".
   - **Ratio 2–10** — clear cache effect. Small buffer served from L1 cache, large buffer hits RAM. Report PASS: "cache present, ratio=X.Y".
   - **Ratio ≈ buffer_size_ratio (64:1 in this case)** — no cache effect; both reads hit RAM at similar speed per byte. Report FAIL: "cache absent or disabled".
   - **Ratio < 1** — physically impossible with working timing; report WARN: "measurement_anomaly".

### Buffer sizing

- `DIAG_CACHE_SMALL_BYTES = 2048` — fits comfortably inside any L1 (486 has 8 KB unified, Pentium has 8 KB data, even 386 external cache is typically ≥ 8 KB).
- `DIAG_CACHE_LARGE_BYTES = 131072` (128 KB) — exceeds any plausible L1 and most L2 for 486 family.
- Both allocated as `static unsigned char __far` per the codebase's PF-2 far-buffer convention. 128 KB + 2 KB = 130 KB total FAR data added.

### Iteration count

Target 50 ms per timed interval (well under one PIT wrap, so standard `timing_start`/`timing_stop` suffices — no need for `timing_start_long`). On a 486 DX-2 at 66 MHz:

- Small buffer: 50 ms at 66M cycles/sec = 3.3M cycles available. Each read-with-stride-4 is ~3 cycles (L1 hit). So ~1.1M reads, i.e. ~540 traversals of the 2 KB buffer.
- Large buffer: 50 ms at, say, 15 ns per RAM read (66 MHz DRAM) = ~3.3M cycles / 15 ns. Each stride-4 read of a 128 KB buffer has 32,768 elements; traversal is 32k × ~5 cycles = 160k cycles = 2.4 ms per traversal. Can do ~20 traversals in 50 ms.

Calibration: start with 20 traversals for large, 500 for small. If first iteration returns <10 ms, scale up. If > 100 ms, scale down. Cap at 2000 small / 100 large.

### Pre-requisites on detect_cache

`detect_cache` emits `cache.present=yes|no` based on class-inference. If `cache.present=no` (8088/V20 class), diag_cache skips with verdict N/A and `diagnose.cache.status=skipped_no_cache_expected`. If `cache.present=yes`, diag_cache runs the stride test.

### INI rows emitted

```ini
[diagnose]
cache.status=pass | fail | warn | skipped
cache.small_buffer_us=<elapsed_us for small-buffer run>
cache.large_buffer_us=<elapsed_us for large-buffer run>
cache.stride_ratio=<t_large / t_small formatted as "X.YZ">
```

Verdict: PASS if ratio ≥ 2.0, FAIL if ratio ≤ 1.2, WARN otherwise.

## Design: `diag_dma`

### Intent

Verify that the 8237 DMA controller chips (master at 0x00-0x0F for channels 0-3, slave at 0xC0-0xDF for channels 4-7) respond to register writes. A stuck DMA chip is relatively rare but has been observed in the wild on aging hardware (oxidized socket, bad capacitor in DMA power rail). Catches:

- **DMA controller totally dead** — writes don't read back. Common pre-death symptom of chipset failure.
- **Specific channel dead** — only one channel fails readback. Can happen with partial chipset damage.
- **Slave controller missing** — XT-class systems have only master controller (4 channels). This diagnostic should detect that cleanly without faulting.

### Does NOT detect

- DMA functional correctness during an actual transfer. That would require setting up a real transfer and verifying the result — scope-heavy and potentially destructive (can corrupt memory if misconfigured). Out of scope.
- Cascading channel 4 (master channel 0 cascade pin) — XT cascade logic isn't exercisable without side effects.
- DMA bus-mastering on VLB/PCI — different controller entirely.

### Method: count-register readback

For each DMA channel, the base/count register pair is word-accessed via byte-sequential writes (high/low latched). The simplest non-intrusive probe:

1. Save current count register value (byte-read x2 via the latch mechanism).
2. Write a known test pattern (e.g., 0xA55A).
3. Read it back.
4. Restore the original value.
5. If readback matched the test pattern: channel responsive.
6. If readback matched 0xFFFF or the inverse pattern: channel dead or slave absent (for channels 4-7 on XT).
7. Any other value: anomaly, log for debug.

### DMA port map (PC/AT)

Master (channels 0-3):
- Base address register: 0x00, 0x02, 0x04, 0x06
- Word count register:  0x01, 0x03, 0x05, 0x07
- Mode register: 0x0B
- Clear byte-pointer flip-flop: 0x0C

Slave (channels 4-7):
- Base address register: 0xC0, 0xC4, 0xC8, 0xCC (16-bit aligned)
- Word count register:   0xC2, 0xC6, 0xCA, 0xCE
- Mode register: 0xD6
- Clear byte-pointer flip-flop: 0xD8

For probe: use the word-count register (reading it is non-destructive — it's a mirror of the 16-bit count value). Write 0xA55A, read back. Expect 0xA55A (or close to it, accounting for 16-bit latch semantics).

### Channel mask considerations

The 8237 DMA controller has a mask register (port 0x0A for master, 0xD4 for slave) that disables individual channels. During probe, DO NOT touch the mask register — we want to observe whatever state the BIOS/DOS has set up. Specifically: do not unmask a masked channel, and do not mask an unmasked channel. The probe only reads+writes the count register of each channel.

### Pre-flight safety

- Channel 0 (master ch 0) is DRAM refresh on AT-class machines. **SKIP channel 0 from probing** — writing to its count register even briefly could cause a refresh hiccup that corrupts memory. XT-class machines use channel 0 for a different purpose (also safety-critical). Never probe channel 0.
- Channel 4 (slave ch 0) is the cascade link on AT. Writing to its count is harmless in theory but cascades through to channel 0. **SKIP channel 4** for the same paranoia.
- Probe channels 1, 2, 3, 5, 6, 7 only.

### INI rows emitted

```ini
[diagnose]
dma.ch1_status=pass | fail | skipped
dma.ch2_status=pass | fail | skipped
dma.ch3_status=pass | fail | skipped
dma.ch5_status=pass | fail | skipped  (XT machines: skipped_no_slave)
dma.ch6_status=pass | fail | skipped
dma.ch7_status=pass | fail | skipped
dma.summary=ok (6/6 channels responsive) | partial (N/6 channels responsive) | dead (0/6 channels responsive)
```

Verdict on `dma.summary`: PASS if all probed channels responded, WARN if any channel failed readback, FAIL if all probed channels failed.

For XT-class machines (detected via `cpu.class=8088|8086|v20|v30` or `bus.class=isa8`), channels 5-7 are skipped with `skipped_no_slave` — no slave controller present.

### Host-testable pure-math kernel

```c
/* Given: per-channel pass/fail/skip array + probe count. Produce summary verdict. */
verdict_t diag_dma_summary_verdict(int ch_pass, int ch_fail, int ch_skip);
```

Host tests: 6-pass (all PASS), 3-pass-3-skip (AT-mode partial), 5-pass-1-fail (one channel dead = WARN), 0-pass-6-fail (controller dead = FAIL).
