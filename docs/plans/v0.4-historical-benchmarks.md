# v0.4 Historical Benchmarks — Dhrystone, Whetstone, PC-XT Ratings

> **Status:** Plan document. Drafted alongside initial implementation (Arc 2, 2026-04-18 evening). Ratifies decisions made during that sprint so future phase re-plans start from a known baseline.

## Scope

Adds three DOS-scene-recognized benchmarks and their derived metrics to CERBERUS:

1. **Dhrystone 2.1** — integer throughput. Standard metric: Dhrystones/second.
2. **Whetstone** — floating-point throughput. Standard metric: K-Whetstones/second.
3. **PC-XT relative ratings** — each of CPU / FPU / memory expressed as × IBM PC-XT.

Plus a new consistency rule (Rule 10, `whetstone_fpu_consistency`) exercising the cross-check between Whetstone's completion state and the FPU-detection report.

See [`checkit-comparison.md`](checkit-comparison.md) for why these three in particular and how they fit the broader CheckIt-comparability story.

## Sources

### Dhrystone 2.1

- Reference implementation: Reinhold Weicker, *"Dhrystone Benchmark: Rationale for Version 2 and Measurement Rules"*, ACM SIGPLAN Notices, August 1988.
- Public-domain C source widely mirrored (e.g., [netlib](https://www.netlib.org/performance/html/dhrystone.html)). License: free use with attribution; Weicker explicitly placed it in the public domain for research.
- Our port stays faithful to the 2.1 semantics (not 2.0, not 2.2 — 2.1 is the canonical scene reference).

### Whetstone

- Original: Harold Curnow, *"A synthetic benchmark"*, The Computer Journal 19(1), 1976.
- C port: derived from the Fortran original, most widely circulated form is the Painter/Weicker cleanup. Public domain.

### PC-XT baseline

Published reference numbers for a stock 4.77 MHz IBM 5160 (PC-XT):

| Axis | PC-XT reference | Source |
|---|---|---|
| Dhrystones/sec | ~346 | BYTE 1988 benchmark round-up; later re-measured in PC Magazine 1991 archive. |
| K-Whetstones/sec | ~0.080 (no 8087 assumed) | CheckIt 3.0 documentation; matches Curnow's original PC-XT measurement. |
| Memory bandwidth KB/s | ~140 | IBM PC Technical Reference Manual timing tables (1T wait state, 4.77 MHz bus). |

**Baseline decision: use published numbers, not a CERBERUS-measured PC-XT.** Reasons:

- Two of the three published numbers come from sources (CheckIt, PC Magazine) that the DOS scene already treats as authoritative. Adopting them makes CERBERUS directly comparable to published review data from the 1990s without a translation step.
- We don't own a stock 5160 to measure against, and any non-stock 8088 (V20, faster bus, different memory) produces different numbers that would bias our entire ratio scale.
- Documenting the source lineage in-code means any future operator can verify the constants against the named references, which is strictly better than "the number Tony measured on a Tuesday in 2026."

Constants live as `static const unsigned long` in the PC-XT ratio module with source-citation comments. Override mechanism deferred to v0.5 (env var or CLI flag) — for v0.4 the published values are baked in.

## New INI rows

Additive to existing `[bench]` section:

```ini
[bench]
; existing:
cpu.int_elapsed_us=...
cpu.int_iters_per_sec=...
cpu.int_us_per_iter=...
memory.write_kbps=...
memory.copy_kbps=...
memory.read_kbps=...
fpu.elapsed_us=...
fpu.ops_per_sec=...
fpu.us_per_op=...
; new (v0.4):
cpu.dhrystones=<integer dhrystones/sec, from bench_dhrystone>
cpu.dhrystones_elapsed_us=<PIT-measured runtime>
fpu.k_whetstones=<integer k-whetstones/sec, from bench_whetstone>
fpu.whetstone_status=<"ok" | "skipped_no_fpu">
fpu.whetstone_elapsed_us=...
cpu_xt_factor=<float with 2 decimals, = dhrystones / 346>
fpu_xt_factor=<float with 2 decimals, = k_whetstones / 0.080 when whetstone ran, empty when skipped>
mem_xt_factor=<float with 2 decimals, = memory.copy_kbps / 140>
```

All three `*_xt_factor` values are reported as the decimal ratio (not the PC-XT=1.0 formatted string), to keep the INI parseable. UI layer may format as "97.70× PC-XT" for display.

## Consistency rule

### Rule 10: `whetstone_fpu_consistency`

**Slot rationale.** Session-brief language was "Rule 9" but Rule 9 (`rule_8086_implies_isa8`) is already occupied and Rule 8 is reserved in `consist.c`'s block-comment footer for the future `cache_stride_vs_cpuid_leaf2` cross-check. Rule 10 is the next free slot. Same off-by-one-numbering pattern as Rule 7 took; noted in commit body and source comment.

**Input keys.**
- `fpu.detected` — from `detect_fpu`. Absent ⇒ no FPU report.
- `bench.fpu.whetstone_status` — emitted by `bench_whetstone` regardless of FPU presence. Values: `"ok"`, `"skipped_no_fpu"`.
- `bench.fpu.k_whetstones` — present only when `whetstone_status=ok`.

**Verdicts.**

| FPU detected | Whetstone status | Whetstone result | Verdict |
|---|---|---|---|
| present | ok | >0 | PASS (consistent: FPU reported + Whetstone completed with result) |
| present | ok | 0 | WARN (FPU reported but Whetstone didn't produce a number — FPU may be marginal or Whetstone miscalibrated) |
| present | skipped_no_fpu | N/A | FAIL (FPU reported present but Whetstone's FPU probe said absent — detection disagreement) |
| absent | skipped_no_fpu | N/A | PASS (consistent: no FPU, Whetstone correctly skipped) |
| absent | ok | >0 | FAIL (FPU reported absent but Whetstone ran — detection underreported, possible socketed 8087 probe miss) |
| any | any | missing keys | N/A (rule not applicable) |

**Host tests.** Three scenarios at minimum: normal-FPU-present PASS, absent-FPU PASS, detection-disagreement FAIL.

**Existing Rule 5 (`fpu_diag_bench`) is not modified.** Rule 5 cross-checks diag.fpu vs bench.fpu.ops_per_sec (CERBERUS's internal FPU benchmark). Rule 10 cross-checks detect.fpu vs bench.fpu.whetstone_status (the CheckIt-compatible benchmark). Parallel axes on different inputs — both fire, they are not redundant.

## Implementation plan

### Module split

- `src/bench/bench_dhrystone.c` — Dhrystone 2.1 reference port. Exposes `void bench_dhrystone(result_table_t *t, const opts_t *o);`.
- `src/bench/bench_whetstone.c` — Whetstone port with FPU-presence gate. Exposes `void bench_whetstone(result_table_t *t, const opts_t *o);`.
- Existing `src/bench/bench.h` gains both prototypes.
- Existing `src/bench/bench_all.c` gains two calls, wrapped in the same crumb macros as the existing bench paths.

### PC-XT ratio computation

- Lives in `src/bench/bench_all.c` after all individual benches complete, so the ratio math sees all dependent values.
- Emits three `bench.*_xt_factor` rows as `V_STR` with pre-formatted decimal output (`"97.70"`), since the result table's numeric types are integers and we need fractional precision without bringing in float formatting overhead across the rest of the codebase.
- Reference constants declared in a small header-file-like block at the top of `bench_all.c` with source citations inline.

### UI integration (v0.4 scope — bar-graph deferred)

Summary view (`src/core/ui.c::ui_render_summary`) gains four new rows in the `rows[]` static:

```
{ "bench.cpu.dhrystones",       "dhrystones" },
{ "bench.fpu.k_whetstones",     "  whetstone" },
{ "bench.cpu_xt_factor",        "× PC-XT cpu" },
{ "bench.fpu_xt_factor",        "    fpu" },
```

Three-pane bar-graph UI (from `checkit-comparison.md` section 2.4) stays blocked on UI-hang resolution and lands in v0.5.

### Host tests

- `test_consist.c` — three scenarios for Rule 10 (plus existing 30 pass; bring total to 33).
- Benchmarks themselves are not host-tested: they rely on PIT timing and integer/FPU math that differs across emulator and hardware. Their host-side check is DGROUP + "links" only. Real-hardware comparison against CheckIt provides the integration-level correctness check.

## Iteration calibration

### Dhrystone

The reference runs 5000–100000 iterations depending on the target. DOS conventions:

- 8088 at 4.77 MHz: 10000 iterations ≈ 30s. Overkill; 2000 ≈ 6s.
- 486 DX-2 at 66 MHz: 500000 iterations ≈ 5s is a reasonable calibrated target.

Target approach: **auto-calibrate** via a small pre-run that measures ~100ms of execution and scales the main run to a ~5s runtime. Same pattern as `bench_cpu`'s existing iteration-auto-calibration.

Fallback to a fixed iteration count (50000) if auto-calibration produces nonsense (too fast = emulator artifact, too slow = hang risk).

### Whetstone

Similar auto-calibrate pattern. K-Whetstones are commonly measured over 1-million-loop batches. Target 5s runtime on detected CPU class; fallback 1000 passes.

### DGROUP impact estimate (pre-build)

Per the 52,000-byte ceiling vs current 48,688 (headroom 3,312 bytes):

- Dhrystone reference implementation uses several `Str_Type` static records (~80 bytes) and a handful of `char [30]` strings. Estimated ~400 bytes of CONST / BSS added.
- Whetstone uses ~20 `float` constants (80 bytes) and one module-scope PA array used by the Dhrystone-similar record manipulation (~200 bytes).
- PC-XT reference constants: 3× `unsigned long` + a shared reference-name string. ~30 bytes.
- New INI rows go through `report_add_str` / `report_add_u32` → each new key string costs ~20 bytes of CONST.

Total estimated: ~1,300 bytes of DGROUP growth. Well within the 3,312-byte headroom. If the port actually blows budget, trim: kill any Dhrystone printf-output strings (we never print the Dhrystone banner, only emit INI rows), which saves ~500 bytes.

## Deliverables & commit split

Three logical commits per session brief:

1. **`bench/dhrystone` commit** — `bench_dhrystone.c` + `bench.h` proto + `bench_all.c` integration for Dhrystone only + UI row for dhrystones + host-test smoke (if any feasible) + docs/plans reference.
2. **`bench/whetstone` commit** — `bench_whetstone.c` + `bench.h` proto + `bench_all.c` integration for Whetstone + UI row + FPU-absent skip path.
3. **`bench/xt-ratings + Rule 10` commit** — PC-XT ratio math in `bench_all.c` + all three `*_xt_factor` INI emits + UI rows for ratios + `rule_whetstone_fpu_consistency` in `consist.c` + 3 new host-test scenarios.

Each commit independently builds green and passes all non-pre-existing host tests.

## Stop conditions

- DGROUP breaches 52,000 before the third commit → stop, trim string tables, resume or document deferral.
- Host-test regression in a suite other than `test_timing` → stop and diagnose.
- Dhrystone or Whetstone output on 486 diverges >5% from CheckIt reference (33,609 / 11,419.9K) → stop and investigate calibration. Wider divergence almost certainly means our iteration counter is off or PIT math is wrong.
