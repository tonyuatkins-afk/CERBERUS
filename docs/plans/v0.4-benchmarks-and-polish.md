## 1. Cache bandwidth benchmark (`bench_cache`)

### Intent

Complement `diag_cache`'s binary presence/absence verdict with a KB/s throughput number. Downstream consumers (the consistency engine, the web dashboard, community submissions) need a continuous measurement, not just PASS/WARN/FAIL.

### Method

Same `static __far` buffers as diag_cache (2 KB small, 32 KB large), but timed for throughput rather than ratio:

```
bench.cache.l1_write_kbps    — stride-4 write loop over the 2 KB buffer
bench.cache.l1_read_kbps     — stride-4 read loop over the 2 KB buffer
bench.cache.ram_write_kbps   — stride-4 write loop over the 32 KB buffer
bench.cache.ram_read_kbps    — stride-4 read loop over the 32 KB buffer
```

Four numbers. Inferences fall out naturally:
- `l1_write / ram_write` ≈ per-write cache multiplier
- `l1_read / ram_read` ≈ per-read cache multiplier (same shape as diag_cache's `stride_ratio` but throughput-unit)
- Read vs write comparisons catch write-through (reads fast, writes slow) vs write-back (both fast) caching policy differences between 486 variants.

### Reusing diag_cache buffers

`diag_cache` already declares `small_buf` + `large_buf` at `__far` scope. Rather than duplicate the allocation (128 KB of FAR wasted on duplicate static arrays would push DGROUP past ceiling), `bench_cache` includes `diag_cache.c`'s header-friendly declarations OR gets refactored to a shared module.

Cleanest refactor path: split diag_cache's static buffer storage into a new `src/core/cache_buffers.c` that both diag_cache and bench_cache link against. Both modules call `cache_buffers_small()` / `cache_buffers_large()` to get FAR pointers to the shared regions. Keeps the 34 KB allocation single-sourced.

**DGROUP accounting:** diag_cache's buffers are FAR — they do NOT count against DGROUP. Moving them to cache_buffers.c doesn't change the DGROUP situation either way. Net neutral.

### Overflow bounds

At 486 DX-2 speed, L1 write throughput ~50 MB/s = 51,200 KB/s. L2 or RAM ~20 MB/s. These fit comfortably in `unsigned long` (up to 4.29e9). The kb_per_sec math already rewritten in `b6c179b` handles the microsecond-precision scaling without overflow.

### Rule candidate enabled

`rule_cache_stride_vs_cpuid_leaf2` is reserved in `consist.c`'s footer as Rule 8. With bench_cache's numeric throughput, the rule can compare measured L1 throughput to the CPU-DB-expected band (similar to Rule 4b's `cpu_ipc_bench` pattern). Scope for v0.4 completion, not the rule itself landing — add the bench first, land Rule 8 when data is available.

## 2. Video bandwidth benchmark (`bench_video`)

### Intent

Measure VRAM write throughput, which is a proxy for "how fast will my game's frame-blit run?" Scene-relevant, CheckIt-comparable (the old benchmark had this too), and currently absent from CERBERUS.

### Method

Write a 16 KB block to VRAM (half the CGA / EGA text-mode buffer size, or full VGA 320×200 mode 13h) and time it. Two sub-tests:

```
bench.video.text_write_kbps    — write to the detected text-mode segment
                                  (B000 for MDA/HGC, B800 for CGA/EGA/VGA)
bench.video.mode13h_kbps       — on VGA only: switch to mode 13h, blit
                                  64 KB to A0000, switch back. Skipped on
                                  MDA/CGA/EGA (mode 13h requires VGA).
```

### Pre-flight safety

- The text-mode test overwrites whatever the current text-mode display shows. For a clean bench, save the first 4 KB of VRAM before the write, write the benchmark pattern, measure, then restore. Same save-restore pattern diag_dma uses for the DMA count register.
- Mode 13h switch requires `INT 10h AH=00h AL=13h` (set mode) and reverse. If CERBERUS is running under Windows DOS box or DOSEMU in a window, mode-switching can mangle the host terminal. Gate with `environment.emulator` check — skip mode 13h test in DOSBox-X / other emulators where the probe produces unreliable numbers.

### Rule candidate

`rule_video_bench_modes_available` (reserved in consist.c footer). After bench_video lands, this rule cross-checks that the text-mode benchmark produced a non-zero number AND (on VGA) the mode 13h benchmark produced a non-zero number. Flags a mismatched pair where text-mode-benches-but-mode13h-fails, which catches VGA cards in CGA-only BIOS compatibility mode.

## 3. Issue #4 Whetstone FPU-assembly rework

### Intent

Close the ~100× divergence from CheckIt 11,420 K-Whet reference. Current Whetstone at `-ot -oi` produces 109 K-Whet due to volatile-E1[4] memory traffic dominating per-unit time (see `docs/plans/checkit-comparison.md` §6 for the full iteration log).

### Method

Replace the inner bodies of `PA()`, `P0()`, and `Module 2` with NASM-compiled 8087 kernels. The outer `run_whetstone_units` stays in C for iteration control + module orchestration; only the hot inner loops move to assembly.

```
src/bench/bench_whetstone_a.asm    (new file)

_PA_asm  — takes FAR pointer to E[4] (non-volatile, asm owns the scope),
           iterates 6 times, uses x87 stack for intermediate values, only
           writes final E[0..3] back to memory once per call.
_P0_asm  — takes FAR pointer to E[4] + J/K/L indices, shuffles array
           slots without intermediate FST.
_mod2_asm — Module 2's 12-iteration inner loop, entirely register-resident.
```

Volatile semantics preserved at the caller level: `E1[4]` remains `volatile` at file scope in bench_whetstone.c. The asm kernels take a non-volatile FAR pointer — asm owns the access pattern and the volatile barrier is maintained by the caller's checksum read after the whole run, same as the current C structure.

### Acceptance

- BEK-V409 real-iron Whetstone within ±5% of 11,420 K-Whet.
- `bench.fpu.k_whetstones` confidence upgrades CONF_LOW → CONF_HIGH.
- `bench.fpu_xt_factor` likewise CONF_HIGH.
- Existing host-test Scenarios CC/DD/EE/FF/GG in test_consist.c remain green (Rule 10 logic is scale-invariant).
- Whetstone inner kernels have cycle-count estimates in source comments against the 486 DX-2 reference (so future 486 variants' deviations are quantifiable).

### Risk

NASM 8087 assembly is the first time CERBERUS has hand-rolled x87 code. Watcom + NASM interop at the C-callable boundary needs care — calling convention, register saves, and the `__far` pointer ABI. Low-level work but well-understood pattern (many reference DOS benchmarks of the era did this).

## 4. Rule 11: `dma_class_coherence`

### Intent

XT-class CPUs (8088/V20/V30) ship with only the master DMA controller; no slave at 0xC0-0xDF. If `cpu.class` reports XT-class AND `diagnose.dma.ch5_status=pass` (or any slave channel responsive), the DMA detection path and the CPU detection path disagree.

### Method

```c
static void rule_dma_class_coherence(result_table_t *t)
{
    const result_t *cls = find_key(t, "cpu.class");
    const result_t *ch5 = find_key(t, "diagnose.dma.ch5_status");
    const char *cv = key_value(cls);
    const char *c5 = key_value(ch5);
    int is_xt;

    if (!cv || !c5) return;
    is_xt = (strcmp(cv, "8088") == 0 || strcmp(cv, "8086") == 0 ||
             strcmp(cv, "v20")  == 0 || strcmp(cv, "v30")  == 0);
    if (!is_xt) return;
    if (strcmp(c5, "skipped_no_slave (XT-class machine)") == 0) {
        report_add_str(t, "consistency.dma_class_coherence",
                       "pass (XT-class CPU with no DMA slave, as expected)",
                       CONF_HIGH, VERDICT_PASS);
    } else if (strcmp(c5, "pass") == 0) {
        report_add_str(t, "consistency.dma_class_coherence",
                       "WARN: XT-class CPU reported, but DMA slave channel responsive — detection contradiction",
                       CONF_HIGH, VERDICT_WARN);
    }
}
```

### Rule-numbering check

Rule slots today: 1, 2, 3, 4a, 4b, 5, 6, 7, 9, 10. Rule 8 reserved for `cache_stride_vs_cpuid_leaf2`. Rule 11 is the next free slot; this rule takes it.

### Host-test scenarios

- XT-class + dma_ch5=skipped → PASS
- XT-class + dma_ch5=pass → WARN (contradiction)
- AT-class + dma_ch5=pass → no-op (rule doesn't apply)
- AT-class + dma_ch5=skipped → no-op (safety-skip path)

## 5. Rule 4b narration extension

### Intent

Today Rule 4b's WARN message lists three possible causes ambiguously: "bench below expected (throttle, cache disabled, or TSR stealing cycles)". With diag_cache now live, the rule can eliminate one branch of that ambiguity when the cache diagnostic produces a verdict.

### Method

In `rule_cpu_ipc_bench`:

- If `bench.cpu.int_iters_per_sec < bench_iters_low` AND `diagnose.cache.status=pass` AND `diagnose.cache.status` message contains "cache_working" → narration: "bench below expected; cache is working so likely TSR or thermal throttle".
- If `bench.cpu.int_iters_per_sec < bench_iters_low` AND `diagnose.cache.status=fail` → narration: "bench below expected AND cache diagnostic FAIL — cache disabled in BIOS or absent".
- If `bench.cpu.int_iters_per_sec < bench_iters_low` AND no cache diagnostic → existing ambiguous narration.

No new rule number. The existing Rule 4b just gets smarter at explaining WHY the WARN fired.

### DGROUP ceiling decision

The 52,000 ceiling was a session-level soft limit picked during v0.2-rc1 work to preserve maneuvering room. The hardware-level ceiling is 65,536. A v0.4 that lands four new modules plus two rule enhancements is a reasonable time to lift the soft limit.

**Recommended new ceiling: 56,000.** Leaves ~9,500 bytes between the expected v0.4 DGROUP (~52,900) and the hardware ceiling (65,536). v0.5 (NetISA upload + bar-graph UI) can comfortably fit.

If the ceiling decision is contentious: revisit in the v0.5 re-plan; for v0.4 implementation, track actual DGROUP post-build and report before tagging.
